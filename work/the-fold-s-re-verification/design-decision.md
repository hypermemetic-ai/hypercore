# design-it-twice: where the re-verify timeout-vs-failure distinction lives [machine]

A load-bearing interface — WHERE a re-verify subprocess's outcome (completed / overran the time bound /
could not run) is distinguished, and how that distinction composes with `scenario.gate`,
`scenario.reverify`, and `delta.fold` — was designed twice, three candidates briefed to place the seam
at radically different altitudes (the OS boundary / the keystone's return contract / the fold's reason
string), each compared on depth, locality, and seam placement. It is load-bearing because the bug is
exactly a misplaced seam: today the distinction is made nowhere — `_run_merged`/`_run_at` catch
`except (OSError, subprocess.SubprocessError): return None`, collapsing a `TimeoutExpired` (a perf
overrun) and a real could-not-run into one `None` that becomes one misleading "could not be re-verified"
refusal that discards a verified ~20-minute build. Wherever the seam lands, every caller of the re-verify
inherits whether a slow machine can be told from a broken build.

design-decision: where the re-verify timeout-vs-failure distinction lives → a typed run outcome at the capped-subprocess seam (`_capped_run`) — the single point where `subprocess.run(timeout=)` raises is the one place the OS itself distinguishes a budget overrun (`TimeoutExpired`) from a could-not-run (`OSError`), so the classification is made there, once, into `completed(code) | overran | could-not-run` and named all the way up through `reverify`/`gate` to `fold`; the conflating `except (OSError, SubprocessError): return None` is deleted in one helper that `_run_merged` and `_run_at` both call, the bound passed in as a parameter (so it scales per-suite in production and is drivable with a small budget in a fixture), the distinction made once where the information exists rather than reconstructed by string-matching a reason at the fold (Candidate C's information leak) or duplicated across both runners (Candidate B's two-sites-must-agree).

## Grounds

- **Candidate A — a typed run outcome at the capped-subprocess seam `_capped_run(args, cwd, env, budget)` (chosen)**: the one helper both `_run_merged` and `_run_at` route through wraps the single `subprocess.run(..., timeout=budget)` call and returns a typed `completed(code) | overran | could-not-run` — the three outcomes the OS already hands it (a return code, a `TimeoutExpired`, an `OSError`). `reverify`/`gate` map the type to their verdict (`completed(0)`→continue, `completed(≠0)`→red-refuse, `could-not-run`→the could-not-re-verify refusal, `overran`→a distinct resource-limit signal retried once then surfaced as a retryable decision); `fold` branches the resource-limit signal away from `CannotFold`-broken. **Deepest**: a lot of behavior — three-way classification, the scaling per-suite budget, the propagation that keeps a perf blip from discarding a build — hides behind a tiny typed return. **Most local**: the conflating `except (OSError, SubprocessError): return None` is replaced in exactly one place; `_run_merged`/`_run_at` shrink to call it; nothing reconstructs the distinction downstream. **Seam placement is right**: the distinction is born at the OS boundary where `TimeoutExpired` and `OSError` are already separate exceptions, so naming them costs nothing and loses no information — it *defines the error out of existence*, turning a timeout from an exception-that-looks-like-failure into a named normal outcome. The deletion test passes: the two conflating `except` clauses and the fixed `timeout=180` literal both delete with nothing left behind. Its one cost — the classification verb the gated scenario drives needs the new `_capped_run` seam, absent at the fork base, so the gate's base-red is overdetermined (seam-missing) and the verb's adequacy is the **watched archive-gate residue the architect attests** at integrate (`coherence`), exactly as the re-verify keystone it extends is watched — is honest and already in the system's grain, not a new hole.

- **Candidate B — a typed reason at the `reverify`/`gate` return contract (rejected)**: keep `_run_merged`/`_run_at` returning `int | None` but have each catch `TimeoutExpired` separately and signal it upward, and change `reverify` to return a typed `red | could-not-run | resource-limit` instead of `str | None`. Rejected on **locality and depth**: the distinction is needed at the subprocess, but B reconstructs it one layer up — so **both** `_run_merged` and `_run_at` must independently catch and re-signal the timeout, the same classification written twice in two helpers that must stay in agreement (information leakage — the gravest red flag after shallowness, two places that must change together). It fixes the return contract without fixing where the knowledge lives, leaving the conflation latent in each runner.

- **Candidate C — a structured reason string matched at the `delta.fold` boundary (rejected)**: leave `reverify` returning `str | None` but make the reason string carry a recognizable marker so `fold` can pattern-match a resource-limit reason from a red reason and branch. Rejected as **shallowest and an outright information leak**: the classification would live in the agreement between `reverify`'s wording and `fold`'s matcher — a format two modules must encode and decode identically, the exact temporal/representation coupling depth forbids — and a one-word edit to a reason message would silently re-break the distinction with no type to catch it. It pushes the seam furthest from where the information is born (the OS boundary) to where it is most fragile (a string compare).
