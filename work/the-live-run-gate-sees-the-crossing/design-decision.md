# design-it-twice: where the fenced-run trace is written and read [machine]

A load-bearing interface — **where and how a fenced crossing's run-trace is written and read so it
composes with the existing watched traces without conflation** — was designed twice, candidates
one-mechanism-on-the-existing-seam and split-the-signal, each isolated as its own brief and judged on
depth, locality, and seam placement. It is load-bearing because the trace is the **evidence #11 is
gated on** (the live blocker on `worker-disciplines-become-a-loadable-skill`) and because the existing
watched-evidence seam (`provenance.commit_verdict` / `watched_trace`) is explicitly built to be
capability-agnostic — "the one seam serves the vocabulary check, the depth scan, and every later
watched standard, each naming its own mechanism" — so the decision is whether the fenced-run trace
joins that seam as one more named mechanism or becomes its own parallel signal. The fork is real: the
defect is an **over-broad reader** (`view._has_watched_trace` flips on *any* `*.verdict.md`), so the
fix can either narrow the reader to a new named mechanism or split the gate into two distinct signals.

design-decision: where the fenced-run trace is written and read → one-mechanism-on-the-existing-seam (`integrate` writes a `fenced-run` verdict via `provenance.commit_verdict`; `view._live_status` reads that mechanism by name) — the watched-evidence seam was designed for exactly this ("every later watched standard, each naming its own mechanism"), so a fenced-run trace joins it as one more named mechanism with no new seam; the defect is purely that the reader reads "any verdict.md" instead of naming its mechanism the way every other watched-trace consumer (`provenance.watched_trace(node, mechanism)`) already does, so the fix is to make the reader name `fenced-run` specifically — which fixes the conflation with the vocabulary and depth traces by construction (they carry their own mechanism names) and keeps the live-status render a single honest line; "live-run" is resolved to **fenced**, and the label is reworded to claim only that a fenced crossing folded, never that the unattended autonomy loop is proven. Candidate split-the-signal is rejected because it builds a second readiness line and a second mechanism (`autonomous-run`) that **nothing yet reads** — no autonomous run has occurred and #11 needs only the fenced signal — adding interface for a distinction the system does not yet consume, the opposite of pulling complexity downward; if the operator later wants a distinct "the unattended loop has run" signal, that is a future ask landing a second named mechanism on the *same* seam, not a speculative split built now.

## Grounds

- **one-mechanism-on-the-existing-seam — `integrate` writes a `fenced-run` verdict, the view reads it by name (chosen)**: the writer lands in `communication.integrate`'s success path (after `delta.fold` returns done, on the node it just archived) as `provenance.commit_verdict(node, "fenced-run", <verdict>)` — the same write-and-commit-in-one-act seam the vocabulary check and depth scan already call. The reader, `view._live_status`, stops asking "is there *any* `*.verdict.md`?" and asks "is there a **`fenced-run`** trace?" — exactly the mechanism-named question every other watched-trace consumer asks. Deepest on **seam placement**: it reuses the seam the system already declared capability-agnostic and built for "every later watched standard," so no new mechanism is invented; the trace is the live-run gate's named consumer of an existing trail. Deepest on **locality**: the change is two named sites (the integrate write, the view read) plus the reworded label; `provenance` is untouched (it already supports an arbitrary mechanism label). Deepest on **depth**: it hides "what a fenced run leaves and where the live gate reads it" behind one mechanism name, adding behavior (the gate tracks reality) behind a small interface, and **fixes the conflation by construction** — the vocabulary and depth traces keep their own names, so a named reader can never read them as fenced evidence. The deletion test passes: the over-broad "any verdict.md" read deletes cleanly, replaced by the mechanism-named read. Honest by construction: the live-status label is reworded to say a *fenced crossing* folded (or has not yet), never that the unattended loop is proven, so the render claims exactly what the trace attests.
- **split-the-signal — two distinct signals, fenced-run and autonomous-run, each its own readiness line (rejected)**: keep `worker.run`/integrate writing a `fenced-run` trace **and** add a separate `autonomous-run` trace the scheduler's unattended self-dispatch writes, with `view._live_status` rendering two readiness lines — "fenced-run-trace" (read by #11) and "never-run-autonomous" (the unattended loop, unproven). Rejected on **depth**: it exposes a second signal and a second readiness line for a distinction **nothing yet consumes** — no autonomous run has occurred, and #11's gate needs only the fenced signal — so it is interface built ahead of any reader, speculative generality, the opposite of pulling complexity down. Rejected on **locality**: it additionally touches the scheduler (to write the autonomous trace) and the view's multi-line rendering, widening the change for a signal with no current consumer. Rejected on **seam placement**: it treats "fenced" and "autonomous" as two parallel mechanisms now, when the existing seam already lets a future autonomous signal land as one more named mechanism the day an unattended run actually happens — so the split pays today for a seam the system can add later at no extra cost. It also cannot satisfy this node's own red→green more cheaply than the chosen design, while costing more to build.
