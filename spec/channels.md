# channels
<!-- vision: derived channel, anchor -->

The derived static channels. An agent is grounded through channels rendered from one source — the
worker's depth grounding, the architect's methodology skills, the minimal shared anchor. The
per-episode prompt is *live*, assembled fresh and never stale; the **static** channels are artifacts an
external harness auto-loads, so they cannot be read live — they must be written to disk and regenerated
whenever their source changes. This capability owns that regeneration: the fold re-renders every static
channel from the spec in the same act that lands the spec change, so a committed channel can no more
drift from the source it derives from than the spec can drift from a folded delta. And because a
channel only grounds an agent if the harness actually loads it, the materialized channels must
**conform to the harness's field standard and resolve** — land where the harness reads, meet its
schema, carry pointers that exist — or the fold fails loudly rather than shipping a channel no one reads.

### Requirement: the fold materializes every static channel from its source
Every static channel — each methodology skill, the worker's depth skill, the shared anchor, and the
`CLAUDE.md` bridge — MUST be a pure render of its spec source, regenerated by the fold in the same act
that lands the spec change. None is hand-authored. A committed channel cannot drift from the source it
derives from, because the fold re-renders it; editing a source and folding again re-renders the channel
to match.

#### Scenario: a fold re-renders the channels
- WHEN a fold lands and its source later changes
- THEN the channel is materialized from the current source, and a fold after the source changes
  re-renders it to follow — no committed channel disagrees with its source

  ```check
  plant
  fold
  materialized
  derived
  edit
  fold
  follows
  ```

### Requirement: a methodology skill is single-sourced from its capability slice
A capability's skill MUST render from its `spec/<cap>.md` slice as a progressive-disclosure
`SKILL.md`: the metadata names when to load it, the body carries the slice's overview and its
disciplines (each requirement's statement), the scenarios stay in the slice, and a pointer back to
`spec/<cap>.md` names the single source. The disciplines are the durable "what good looks like",
single-sourced — no second copy of the requirement set to drift.

#### Scenario: the skill renders its slice
- WHEN a methodology skill is rendered
- THEN it is a `SKILL.md` naming its capability, carrying the slice's overview and discipline
  statements but not its scenarios, and pointing back to `spec/<cap>.md` as its single source

  ```check
  plant design-it-twice
  skill-rendered
  single-sourced
  ```

### Requirement: the anchor is the minimal shared agents file, its skills index derived
The shared anchor MUST carry only non-inferable operational content — the check command and the
build/hand-back convention — and nothing inferable: no per-capability requirements, no code, no
identity prose. Its skills index MUST be derived from the channels' own registry, so a newly registered
skill appears by construction. Claude Code reads `CLAUDE.md`, not a bare `AGENTS.md`, so a derived
`CLAUDE.md` bridge imports the anchor (`@AGENTS.md`).

#### Scenario: the anchor is minimal and its index is derived
- WHEN the anchor is rendered
- THEN it carries the check command and the hand-back convention and nothing inferable, its skills
  index lists every registered skill pulled from the registry, and the `CLAUDE.md` bridge imports it

  ```check
  anchor
  minimal
  index-derived
  bridge
  ```

### Requirement: the materialized channels conform to the field standard and resolve
Every materialized channel MUST conform to the harness's field standard — each skill landing where the
harness discovers it (`.claude/skills/<cap>/SKILL.md`), its frontmatter meeting the documented schema
(a name equal to its directory, a non-empty description within the limits) — and every pointer it
carries (`spec/<cap>.md`, `@AGENTS.md`) MUST resolve to a file that exists. A channel that renders
perfectly but lands where no harness reads it, or carries a pointer a rename has broken, MUST fail here
loudly rather than ship silently.

#### Scenario: the channels conform and resolve
- WHEN the channels are materialized
- THEN the pre-fix render (skills in the bare root only, no bridge) fails the discovery standard, the
  live render passes it, every rendered skill's frontmatter conforms, and every pointer resolves

  ```check
  old-render
  discovery-red
  materialize
  discovery-green
  conforms
  resolves
  ```
