## problem

The current phase-one collaboration mechanism still lets the machine privately jump from
the operator's broad intent to a settled route, then ask the operator to sign a frame
after the route is already chosen. Work `005` and `006` tried to harden this by adding
more required frame fields. That produced a category error: the field scanner can verify
non-placeholder text, but it cannot verify that an operator-machine exchange happened or
that the operator's side was real rather than written by the machine.

The structural defect is timing and anchoring. The only current operator act is sign-off,
and sign-off happens after the frame already contains a route. The result is a frame that
can be mechanically complete while still being a machine-authored plan with manufactured
confidence.

This work replaces the 005/006 field pile with a thinner phase-one arc: understanding
before route, scrutiny sized by reversibility, mechanically spawned base review for
one-way work, substantive operator direction before route, then a lean recoverability
frame and sign-off.

## operator deliberation

### problem/domain map

The domain is hypercore's phase-one governance. Current phase one has five loop gates with
one operator act, sign-off. The problem is not lack of prose; it is that the only
operator-anchored mechanism sits after the route is already present. Current checks scan
fields in a one-party artifact, so the machine can author both sides of a supposed
collaboration and pass.

The redesigned domain keeps the existing loop but changes the phase-one shape:

- understanding beat: teach-back, alternative framing, information-gain question, no
  route;
- reversibility classification: one-way work gets heavier scrutiny; two-way work stays
  fast;
- review: one-way work gets a mechanically spawned base roster before route settlement;
- direction: the operator records a substantive route choice, constraint, or delegation;
- frame: the machine writes the route and lean recoverability fields after direction;
- sign-off: the operator attests informed expectation and understanding.

### evidence standard

Evidence depth for this frame is local plus adversarial panel. The operator brief already
contains external research synthesis sized to the problem. The immediate frame decision is
about fitting that design into hypercore's existing intent and material, so local corpus
evidence and independent limited-context review are the load-bearing evidence. No further
web search is needed before sign-off because the unsettled question is mechanism shape,
not the general literature claim that prompted the brief.

### evidence basis

Local evidence:

- `intent/collaboration.md` currently requires recoverable common ground and operator
  deliberation.
- `intent/loop.md` currently requires an operator-deliberation record before sign-off.
- `intent/machine-statements/loop.md`, `adapter/gates/frame.md`, `adapter/loop.sh`, and
  `check.sh` currently require the 005/006 fields.
- `adapter/loop.sh` currently treats `signoff.md` as the only anchored operator act.
- The current scanner reads non-signoff markdown by label/heading and would be too loose
  for route-before-direction enforcement unless tightened.
- Adopted histories `005` and `006` show the accretion path and why this work should be a
  coherent replacement, not an additive patch.

Panel evidence:

- Soundness review flagged empty direction, optional reviewer dilution, and two-way
  misclassification as bypass risks.
- Simplicity review warned that review mechanization can become new ceremony; the route
  keeps the base roster small and avoids quality-scanning claims.
- Contract review found no vocabulary blocker if direction/review remain phase-one acts
  or artifacts, not new gates.
- Red-team review flagged retrospective direction, loose frame parsing, false reviewer
  independence, classification bypass, and phase-two starvation; the route responds with
  substantive direction, direct-before-route refusal, strict parsing, base-flag
  escalation, one-way default criteria, and frame-carried acceptance/excluded
  interpretation.

### options and tradeoffs

Route A: artifact-only review.

This keeps `loop.sh` small and fast, but leaves too much of the review step to machine
posture. It does not satisfy the operator's direction that the review step itself be
mechanical.

Route B: small mechanical base roster. Selected.

This adds `review` mechanics to the adapter, but constrains them: small base roster,
read-only independent sessions, structured verdicts, optional additions advisory only,
and checks that prove the command and artifacts exist without claiming review quality is
proved.

Route C: large reviewer orchestrator.

This overfits the problem, risks slow ceremony, and overclaims independence and model
diversity. It is rejected.

### operator expectation

The operator should expect this work to make the machine stop at a real phase-one
decision surface before route settlement. After adoption, one-way work should not be
signable until the operator has given substantive direction and the base review roster has
produced a review artifact. Small reversible work should remain fast because two-way work
skips the panel unless requested.

### rejection conditions

Shelve or stop if implementation cannot keep the mechanical base review small, if `review`
spawning requires brittle or unavailable Codex capabilities, if strict parsing cannot
reliably distinguish route content from decision/review text, if `direction` cannot be made
substantive without the machine writing the operator's act, or if checks still require the
005/006 field pile after the new act/artifact contract is added.

### unresolved discomfort or open judgement

Open judgement: mechanical reviewer independence is limited by the available harness. The
route must say this plainly and avoid treating a spawned base roster as proof of real
diversity. If the available Codex CLI cannot run independent read-only reviewer sessions,
the panel layer must be surfaced before adoption rather than faked.
